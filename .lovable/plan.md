

# Add Manual "Add Fee" Action to Fee Ledger Card

## Problem

Operators cannot create ad-hoc fee charges from the Payment Operations page. If a fee needs to be logged that wasn't auto-generated by a booking policy or webhook (e.g., a damage fee, a custom late charge), they have no way to create the pending ledger record without touching the Appointment Detail Sheet.

The `appointment_fee_charges` table requires `appointment_id` (NOT NULL, FK to `phorest_appointments`), so every fee must be tied to an appointment — but `policy_id` is nullable, which already supports non-policy-driven charges. The "Add Fee" action will let operators search for an appointment by client name, then attach an ad-hoc fee with a custom amount and description.

## Changes

### 1. `src/pages/dashboard/admin/PaymentOps.tsx` — FeeLedgerCard

**Add an "Add Fee" button** in the card header (right side, next to the pending badge), using `tokens.button.cardAction`.

**Add an "Add Fee" dialog** (standard `AlertDialog` or a `Dialog` for richer form) with:
- **Appointment search**: Text input that queries `phorest_appointments` by `client_name` (ilike), scoped to `organization_id`, limited to recent appointments (last 90 days). Results shown in a simple dropdown/list for selection.
- **Fee type**: Select dropdown — "Manual" pre-selected, with options for deposit/no_show/cancellation/manual.
- **Amount**: Number input (required, > 0).
- **Description/notes**: Optional text input stored as `collected_via = 'manual'` to distinguish origin.
- All fields validated before submit.

**Insert mutation**: Direct insert into `appointment_fee_charges` with:
- `organization_id`: from context
- `appointment_id`: from selected appointment
- `fee_type`: from form (default "manual")
- `fee_amount`: from form
- `status`: "pending" (default)
- `collected_via`: null (not yet collected)
- `policy_id`: null (ad-hoc, no policy)

On success: invalidate `['fee-ledger']` and `['fee-ledger-pending-count']`, toast confirmation, close dialog.

### 2. State additions

`addFeeDialogOpen`, `addFeeForm` (appointment search query, selected appointment, fee type, amount), `appointmentSearchResults`.

### 3. No other files affected

The new pending record appears in the existing table, and the Collect/Waive actions already work on any pending row regardless of origin.

## Technical notes

- Appointment search uses debounced ilike query on `client_name` field
- The `fee_type = 'manual'` label already exists in `FEE_TYPE_LABELS`
- No migration needed — `policy_id` is already nullable, `fee_type` is a free-text column
- No edge function needed — direct Supabase insert from the client (RLS permits org members)
